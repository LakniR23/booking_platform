1. ARCHITECTURE
------------------------------------------------------------

1.1 High-Level Architecture
----------------------------
The application follows a modular monolith architecture built with NestJS.
It is divided into feature modules, each encapsulating its own
controllers, services, and DTOs.

  Client (Web / API Consumer)
       |
       |  HTTP/REST
       v
  +--------------------------------------------------+
  |              NestJS Application                   |
  |                                                  |
  |  +-----------+  +-------------+  +------------+  |
  |  |   Auth    |  |  Services   |  |  Bookings  |  |
  |  |  Module   |  |   Module    |  |   Module   |  |
  |  +-----------+  +-------------+  +------------+  |
  |        |              |                |         |
  |        v              v                v         |
  |  +--------------------------------------------------+
  |  |              Common Layer                       |
  |  |  - Global Validation Pipe                        |
  |  |  - Global Exception Filter (HttpException +       |
  |  |    Prisma error mapping)                          |
  |  +--------------------------------------------------+
  |        |
  |        v
  |  +--------------------------------------------------+
  |  |           PrismaService (ORM)                     |
  |  |  - Single shared PrismaClient instance            |
  |  +--------------------------------------------------+
  |        |
  |        v
  |  +--------------------------------------------------+
  |  |              PostgreSQL                           |
  |  |  Tables: User, Service, Booking                   |
  |  +--------------------------------------------------+
  |
  v
  Swagger UI (/api/docs)


1.2 Module Boundaries
---------------------
- **AppModule**: Root module. Imports ConfigModule (global),
  PrismaModule, AuthModule, ServicesModule, BookingsModule.
  Registers global pipes and filters.

- **PrismaModule**: Provides PrismaService as a singleton
  for database access.

- **AuthModule**: Handles user registration, login, and
  token refresh. Uses JwtStrategy with Passport and
  JwtAuthGuard for protected routes.

- **ServicesModule**: Exposes CRUD endpoints for services.
  Creation, update, and deletion require JWT authentication.

- **BookingsModule**: Exposes booking endpoints.
  Public endpoints: create booking, cancel booking.
  Protected endpoints: list, get by ID, update status.

- **Common Filters**: AllExceptionsFilter standardizes
  error responses and maps Prisma errors to HTTP status codes.


1.3 Request Flow
-----------------
1. HTTP Request arrives at NestJS.
2. Global ValidationPipe sanitizes and validates DTOs
   (whitelist, forbidNonWhitelisted, transform).
3. Route handler in Controller receives request.
4. Guards (e.g., JwtAuthGuard) enforce authentication.
5. Service layer executes business logic using PrismaService.
6. Response returned to client.
7. Unhandled errors caught by AllExceptionsFilter,
   which returns a consistent JSON error format.


1.4 Authentication Flow
------------------------
- **Register**: User submits name, email, password.
  Password is hashed with bcrypt (cost 10).
  User record created with default role CUSTOMER.

- **Login**: User submits email and password.
  Credentials verified against stored bcrypt hash.
  On success, two tokens are issued:
    - access_token (short-lived, signed with JWT_SECRET)
    - refresh_token (longer-lived, signed with JWT_REFRESH_SECRET)

- **Protected Routes**: Client sends
  "Authorization: Bearer <access_token>".
  JwtStrategy extracts and verifies the token.
  Payload attached to request.user.

- **Refresh**: Client sends refresh_token to /auth/refresh.
  Server verifies token, looks up user, issues new pair.


1.5 Database Relations (Prisma Schema)
---------------------------------------
- **User** (1) ----< (*) **Booking**
  - Users authenticate to manage services.
  - Bookings reference customers by name/email/phone,
    not by User ID, because public booking creation
    does not require authentication.

- **Service** (1) ----< (*) **Booking**
  - A service can have many bookings.
  - ON DELETE CASCADE: deleting a service removes
    its bookings.

- **Enums**:
  - Role: CUSTOMER, ADMIN
  - BookingStatus: PENDING, CONFIRMED, CANCELLED, COMPLETED

- **Unique Constraint** on Booking:
  (serviceId, bookingDate, bookingTime)
  Prevents duplicate bookings at the database level.


1.6 Tech-Choice Rationale
--------------------------
- **NestJS**: Provides built-in dependency injection,
  decorators, guards, pipes, and modules for clean
  separation of concerns.
- **TypeScript**: Static typing for maintainability
  and better IDE support.
- **PostgreSQL**: Relational database with strong
  consistency, suitable for booking data.
- **Prisma 7**: Type-safe ORM with auto-generated client,
  migrations, and excellent TypeScript integration.
- **JWT + Passport**: Industry-standard stateless auth.
- **class-validator**: Declarative DTO validation
  integrated with NestJS pipes.
- **Swagger**: Auto-generated interactive API docs
  from decorators.
- **Docker**: Containerized deployment for consistency
  across environments.

2. BUSINESS RULES
------------------------------------------------------------

Rule 1: A booking must belong to an existing service.
-----------------------------------------------------
Enforced in: bookings.service.ts (lines 16-21)
The create() method queries the Service table by serviceId.
If no service is found, a NotFoundException is thrown.

Rule 2: Booking dates cannot be in the past.
--------------------------------------------
Enforced in: bookings.service.ts (lines 24-26)
The service constructs a Date from bookingDate and bookingTime.
If the resulting datetime is before the current time,
a BadRequestException is thrown.
Additionally, the DTO validates the date format (YYYY-MM-DD)
via regex in create-booking.dto.ts (line 28).

Rule 3: Cancelled bookings cannot be marked as completed.
---------------------------------------------------------
Enforced in: bookings.service.ts (lines 114-115)
The updateStatus() method checks if the current status is
CANCELLED and the requested new status is COMPLETED.
If so, a BadRequestException is thrown.

Rule 4: Only authenticated users can manage services.
-----------------------------------------------------
Enforced in: services.controller.ts (lines 14, 34, 43)
POST, PATCH, and DELETE endpoints use the JwtAuthGuard.
Unauthenticated requests receive a 401 Unauthorized response.

Rule 5: Customers can create bookings without authentication.
------------------------------------------------------------
Enforced in: bookings.controller.ts (lines 14-17)
The POST /bookings endpoint is public (no guard applied),
allowing unauthenticated users to create bookings.

Rule 6: Prevent duplicate bookings for the same service, date, and time.
-------------------------------------------------------------------------
Enforced at two levels:
1. Database schema.prisma (line 55):
   @@unique([serviceId, bookingDate, bookingTime])
   Prisma generates a unique constraint in PostgreSQL.

2. Business logic bookings.service.ts (lines 30-39):
   Before creating a booking, the service queries for an
   existing booking with the same serviceId, bookingDate,
   and bookingTime. If found, a ConflictException is thrown.

Additional Rules
----------------
- Service soft-delete via isActive: Services can be
  deactivated (isActive = false) without losing historical
  booking data.
- Booking cancellation is public:
  PATCH /bookings/:id/cancel has no auth guard, allowing
  customers to cancel without an account.
- Pagination, search, and status filtering:
  GET /bookings supports query parameters page, limit,
  status, serviceId, date, and search (customer name/email).

3. SECURITY
------------------------------------------------------------

3.1 Authentication & Authorization
------------------------------------
- JWT-based stateless authentication.
- Access tokens signed with JWT_SECRET.
- Refresh tokens signed with a separate JWT_REFRESH_SECRET.
- Tokens transmitted only over HTTPS in production.
- JwtAuthGuard protects mutation endpoints.
- Passport JWT strategy validates tokens and attaches
  payload to request.user.

3.2 Password Storage
---------------------
- Passwords hashed with bcrypt (cost factor 10).
- Raw passwords never stored or logged.
- bcrypt.mock in unit tests prevents accidental
  exposure of hashing logic details.

3.3 Input Validation
---------------------
- Global ValidationPipe with:
  - whitelist: true (strips non-DTO properties)
  - forbidNonWhitelisted: true (rejects unknown fields)
  - transform: true (auto-converts types)
- DTOs use class-validator decorators:
  - IsString, IsNotEmpty, IsEmail, IsNumber, IsBoolean
  - IsEnum (BookingStatus)
  - Matches (date/time format regex)
  - Min (price >= 0, duration >= 1)

3.4 Error Handling
------------------
- AllExceptionsFilter catches all errors.
- Prisma.PrismaClientKnownRequestError mapped:
  - P2002 (unique violation) -> 409 Conflict
  - P2025 (record not found) -> 404 Not Found
  - P2003 (foreign key violation) -> 400 Bad Request
- PrismaClientValidationError -> 400 Bad Request
- HttpException subclasses returned with proper status codes.
- Error responses are sanitized; no stack traces or internal
  details exposed to clients.

3.5 CORS
--------
- app.enableCors() in main.ts allows cross-origin requests.
- In production, restrict origins via CORS configuration.

3.6 Environment Variables
--------------------------
- Sensitive values stored in .env (never committed):
  - DATABASE_URL
  - JWT_SECRET
  - JWT_REFRESH_SECRET
- .env.example committed as a template.
- ConfigService reads env vars safely.
- Docker Compose passes env vars via environment block.

3.7 Database Security
----------------------
- Prisma uses parameterized queries; SQL injection prevented.
- ON DELETE CASCADE maintains referential integrity.
- UUID primary keys reduce enumeration risk.
- isActive flag allows soft-disable without data loss.

3.8 Secrets Management
-----------------------
- JWT secrets must be strong random strings in production.
- Refresh tokens have separate secret from access tokens.
- Secrets injected via environment variables.
- No hardcoded credentials in source code.

4. TESTING
------------------------------------------------------------

4.1 Backend Unit Tests (Jest)
------------------------------
Location: backend/src/**/*.spec.ts
Command: cd backend && npm test

Covered modules:
- AuthService (auth.service.spec.ts)
  - register: conflict on duplicate email
  - login: invalid credentials, success path
  - refresh: invalid/expired token handling

- AuthController (auth.controller.spec.ts)
  - register endpoint
  - login endpoint

- ServicesController (services.controller.spec.ts)
  - CRUD endpoint contracts

- ServicesService (services.service.spec.ts)
  - create, findAll, findOne, update, remove
  - NotFoundException on missing service

- BookingsController (bookings.controller.spec.ts)
  - endpoint routing and guards

- BookingsService (bookings.service.spec.ts)
  - create: service not found, past date, duplicate, success
  - findAll: pagination, filtering, search
  - findOne: not found
  - updateStatus: cancelled -> completed restriction
  - cancel: status update

- AppController (app.controller.spec.ts)

Test strategy:
- PrismaService mocked to avoid database dependency.
- bcrypt mocked for deterministic password tests.
- Jest fake timers used for date-based assertions.
- Each test file covers success and error paths.

4.2 Frontend Unit Tests (Jest + React Testing Library)
-------------------------------------------------------
Location: frontend/src/tests/*.test.tsx
Command: cd frontend && npm test

Covered pages:
- Bookings.test.tsx
  - renders booking list
  - pagination controls
  - status filtering
  - search input
  - booking creation modal

- Home.test.tsx
  - landing page content

- Login.test.tsx
  - login form rendering and submission

- Register.test.tsx
  - registration form rendering and submission

Test strategy:
- API calls mocked with jest.mock().
- MemoryRouter used for routing context.
- QueryClientProvider wraps components for React Query.
- User interactions simulated with user-event.

4.3 End-to-End Tests (Jest + Supertest)
----------------------------------------
Location: backend/test/app.e2e-spec.ts
Command: cd backend && npm run test:e2e

These tests spin up the NestJS app against a test database
and verify full request/response cycles.

4.4 Coverage
------------
- Backend: npm run test:cov
  Generates coverage report in coverage/ directory.
- Frontend: coverage via Jest configuration.

4.5 Continuous Integration (Recommended)
-----------------------------------------
Add to CI pipeline:
- npm run lint
- npm run test
- npm run test:cov
- npx prisma validate
- docker build checks
